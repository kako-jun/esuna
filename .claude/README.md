## 📖 ドキュメントの使い方

### 初めて開発に参加する場合
1. **[development-guide.md](./development-guide.md)** を読む
   - 開発環境をセットアップ
   - コーディング規約を確認
2. **[architecture.md](./architecture.md)** を読む
   - システム全体の設計を理解
3. **[todo.md](./todo.md)** から取り組むタスクを選ぶ

### 機能を追加する場合
1. **[todo.md](./todo.md)** にタスクを追加（まだなければ）
2. **[development-guide.md](./development-guide.md)** の「新機能の追加方法」に従う
3. **[architecture.md](./architecture.md)** を参考に設計
4. 実装後、**[implementation-progress.md](./implementation-progress.md)** を更新

### 問題が発生した場合
1. **[troubleshooting.md](./troubleshooting.md)** で解決方法を探す
2. 見つからなければGitHub Issueで報告

### ロードマップを確認する場合
1. **[future-features.md](./future-features.md)** で将来の機能を確認
2. **[todo.md](./todo.md)** で具体的なタスクを確認
3. **[implementation-progress.md](./implementation-progress.md)** で現在の進捗を確認

## 🔄 ドキュメントの更新ルール

### いつ更新するか

| ドキュメント | 更新タイミング |
|--------------|----------------|
| architecture.md | 設計が変更されたとき |
| todo.md | タスクが追加・完了・変更されたとき |
| development-guide.md | 開発手順が変更されたとき |
| troubleshooting.md | 新しい問題と解決方法が見つかったとき |
| future-features.md | 新機能のアイデアが出たとき |
| implementation-progress.md | 機能実装が完了したとき |

### 誰が更新するか
- **基本**: 変更を加えた開発者
- **レビュー**: プロジェクトリーダー

### どう更新するか
1. 該当するドキュメントを編集
2. 変更内容をコミット
3. プルリクエストを作成（必要に応じて）

## 🎯 ドキュメントの目的

これらのドキュメントは以下を目的としています：

1. **開発の継続性**: 誰でもプロジェクトを引き継げる
2. **品質の維持**: 一貫した設計と実装
3. **効率的な協働**: ルールを明確にして迷わない
4. **知識の共有**: 技術的な判断の理由を記録
5. **ユーザーサポート**: 問題解決を迅速に
